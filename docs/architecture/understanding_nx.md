> Everything you need to know about Nx and why Project NEXUS uses it.

## What is Nx?

Nx is a **build system and development toolkit** designed specifically for **monorepos** — projects where multiple apps and packages live together in one folder.

Think of it like this. Without Nx, managing a monorepo is like building a house where every room has its own set of tools, its own electricity, and its own set of rules. Nothing talks to each other. Everything is duplicated. You have to manage each room manually.

Nx is the central control system for the house. One place to run everything. One place to configure everything. Smart enough to know which rooms were changed and only fix those ones.

Nx was built by a company called **Nrwl** (now just called Nx) and is used in production by Google, Meta, Shopify, and thousands of other companies. It is the industry standard for managing TypeScript monorepos.

---

## Why NEXUS Uses Nx

NEXUS has two apps (`api` and `web`) and four packages (`database`, `types`, `ui`, `config`). Without Nx, here is what managing that looks like:

- You'd have to manually run the build for each app separately
- You'd have to figure out yourself which order to build things in (database before api, api before web)
- Every time you change a shared type in `packages/types`, you'd have to manually rebuild every app that uses it
- Linting, testing, and formatting would need to be set up and run separately in each app
- There would be no way to know at a glance how everything connects

With Nx:

- One command runs everything
- Nx automatically figures out the build order
- Nx knows which apps are affected by a change and only rebuilds those
- Linting and testing work the same way across every app and package
- You get a visual map of how everything connects

For a solo developer building something as complex as NEXUS, Nx is not optional luxury — it is what keeps the project manageable as it grows.

## The Core Concepts

### 1. The Workspace

The workspace is the entire project folder — `nexus/`. Everything inside it is managed by Nx. The `nx.json` file at the root is the brain. It tells Nx how the workspace is configured, what the default behaviors are, and how caching works.

### 2. Projects

In Nx, every app and every package is called a **project**. Each project has a `project.json` file inside it (generated by Nx) that describes what that project can do. Your projects are:

|Project|Type|Location|
|---|---|---|
|`api`|Application|`apps/api`|
|`web`|Application|`apps/web`|
|`database`|Library|`packages/database`|
|`types`|Library|`packages/types`|
|`ui`|Library|`packages/ui`|
|`config`|Library|`packages/config`|

### 3. Targets

A **target** is a task you can run on a project. Things like `build`, `serve`, `test`, `lint`. Every project has its own set of targets defined in its `project.json`. You run a target like this:

```bash
pnpm nx <target> <project>

# Examples:
pnpm nx serve api        # start the NestJS dev server
pnpm nx serve web        # start the Next.js dev server
pnpm nx build api        # build the NestJS app for production
pnpm nx test api         # run tests for the api
pnpm nx lint web         # lint the web app
```

### 4. Generators

Generators are **code scaffolding tools**. Instead of manually creating files and wiring them up, you run a generator command and Nx creates everything correctly for you.

For example, to add a new `books` module to your NestJS API:

```bash
pnpm nx g @nx/nest:resource books --project=api
```

This single command creates:

- `books.controller.ts`
- `books.service.ts`
- `books.module.ts`
- `books.dto.ts`
- `books.controller.spec.ts` (test file)

All correctly named, all wired together, all following NestJS conventions. You just start filling in the logic.

The `g` is short for `generate`. The `@nx/nest:resource` part means: use the `@nx/nest` plugin and run its `resource` generator.

### 5. The Dependency Graph

This is one of the most powerful features in Nx. It knows exactly how every project depends on every other project. You can visualize it at any time:

```bash
pnpm nx graph
```

This opens a browser window showing a visual map. You'll see that `api` depends on `database` and `types`. You'll see that `web`depends on `types` and `ui`. It updates automatically as you add new imports.

This matters because Nx uses this graph internally to decide what to rebuild and in what order whenever you make changes.

### 6. Caching

This is where Nx becomes genuinely powerful for a solo developer.

Every time you run a task — build, test, lint — Nx stores the result in a **cache**. If you run the same task again without changing anything, Nx skips running it entirely and just replays the cached result instantly. It looks like it ran, but it finished in milliseconds.

The cache is based on the content of your files. Change one line in `apps/api`, and only `api`'s cache is invalidated. The `web`cache stays valid. `packages/database` cache stays valid.

Concretely: building and testing a project that hasn't changed takes **0 seconds**. This matters a lot when your project has 6+ projects and CI/CD pipelines.

The cache lives in `.nx/cache` by default. You never need to manage it manually.

### 7. Affected Commands

Related to the dependency graph and caching, Nx has an `affected` command that only runs tasks on projects that were actually changed by your recent commits:

```bash
pnpm nx affected --target=build
pnpm nx affected --target=test
pnpm nx affected --target=lint
```

For example: you edit a file in `packages/types`. Nx knows that `api` and `web` both import from `types`. So affected will run build/test/lint on `types`, `api`, and `web` — and skip `database`, `ui`, and `config` entirely because they weren't affected.

This becomes critical as the project grows. You never waste time rebuilding things that didn't change.

## What `nx connect` Does

When you run `pnpm nx g @nx/nest:application ...` or similar commands during setup, you may see Nx prompt you with a question about connecting to **Nx Cloud**. The `nx connect` command is what you'd run to do that manually.

**Nx Cloud is an optional remote caching and CI service.** Here is what it does:

- Your local cache lives on your machine. If you run a build, cache it, then go to another computer, the cache is gone. Nx Cloud stores the cache remotely so it's shared.
- In a team setting (three developers like the original proposal), if one developer builds something and caches it, the other two can pull that cache and skip the build entirely.
- It gives you a web dashboard showing build times, test results, and bottlenecks across your team.

**For NEXUS right now, you do not need it.** You are one developer on one machine. The local cache is sufficient. During workspace creation we passed `--nxCloud=skip` for exactly this reason. If you ever want to enable it later it's one command:

```bash
pnpm nx connect
```

It will ask you to create a free account on nx.app and link your repository. The free tier is generous and covers most solo projects. But again — skip it for now. Not needed.

## How Nx is Structured in NEXUS

### `nx.json` — The Master Config

```
nexus/
└── nx.json   ← workspace-wide Nx configuration
```

This file controls:

- Which tasks are cached and which aren't
- What the default base branch is for `affected` commands
- Global generator defaults (so you don't have to pass the same flags every time)
- Named inputs (grouping files together for cache purposes)

You rarely need to edit this manually. Nx manages most of it.

### `project.json` — Per-Project Config

Every project generated by Nx gets its own `project.json`:

```
apps/api/project.json
apps/web/project.json
```

This file defines all the targets for that project — what `build`, `serve`, `test`, `lint` actually do under the hood. For example, the `api`'s `serve` target runs the NestJS dev server. The `web`'s `serve` target runs Next.js. Nx doesn't need to know what technology you're using — it just reads this file.

You will look at these files occasionally but rarely edit them directly in early development.

## The Nx Plugins Used in NEXUS

Nx has a plugin system. Each plugin adds support for a specific technology. NEXUS uses:

|Plugin|What it adds|
|---|---|
|`@nx/nest`|Generators and build targets for NestJS|
|`@nx/next`|Generators and build targets for Next.js|
|`@nx/node`|Generators for generic Node.js projects|
|`@nx/js`|TypeScript/JavaScript library support|

These are installed as dev dependencies in the root `package.json`. When you run a generator like `@nx/nest:resource`, it's using the `@nx/nest` plugin.

## Common Workflows in NEXUS

### Starting Development

```bash
# Start Docker (PostgreSQL + Redis)
docker compose up -d

# Run API and Web simultaneously
pnpm nx run-many --target=serve --projects=api,web --parallel

# Or using the shortcut in package.json
pnpm dev
```

### Adding a New Feature Module to the API

Every major feature in the NestJS API is a **module**. When you need to build the books feature:

```bash
pnpm nx g @nx/nest:resource books --project=api --directory=src/books
```

This scaffolds the entire module. You then fill in the business logic.

### Adding a New Page to the Web App

Next.js uses the App Router, so you mostly just create files in `apps/web/src/app/`. Nx doesn't need to generate these — Next.js handles routing by folder structure.

### Running Tests

```bash
# Test a specific project
pnpm nx test api
pnpm nx test web

# Test everything
pnpm nx run-many --target=test --all

# Test only what changed
pnpm nx affected --target=test
```

### Linting

```bash
# Lint a specific project
pnpm nx lint api

# Lint everything
pnpm nx run-many --target=lint --all
```

### Building for Production

```bash
# Build everything in the right order
pnpm nx run-many --target=build --all

# Build a specific app
pnpm nx build api
pnpm nx build web
```

Nx automatically builds `packages/database` and `packages/types` before building `api`, because it knows `api` depends on them. You never have to think about build order.

### Viewing the Dependency Graph

```bash
pnpm nx graph
```

Opens in your browser. You'll see all 6 projects and the arrows showing which depend on which. As you import `@nexus/types` into more places, the graph updates automatically.

## Module Boundary Rules

One of the most valuable things Nx sets up is **module boundary enforcement**. This is an ESLint rule that prevents you from importing things you shouldn't.

In NEXUS, the rules are:

- `api` can only import from packages tagged `scope:shared`
- `web` can only import from packages tagged `scope:shared` and `scope:ui`
- `packages` can only import from other `packages`

**What this prevents:** Imagine you accidentally import a NestJS-specific thing into your Next.js web app. It would break at build time in a confusing way. Module boundaries catch this immediately with a clear lint error the moment you write the import.

This is especially valuable when you're working alone because you don't have a teammate to catch these mistakes in code review.

## Things Nx Does NOT Do

It's worth being clear about what Nx is not, so there's no confusion:

- **Nx is not a framework.** NestJS is the framework for your API. Next.js is the framework for your web app. Nx just orchestrates them.
- **Nx is not a package manager.** pnpm handles installing packages. Nx just knows how to run tasks.
- **Nx does not write your code.** It scaffolds starting points and manages the build system. The actual library logic, routes, and components are written by you.
- **Nx does not deploy your app.** Deployment is handled by your hosting provider (Railway, VPS, etc.) later.

## The Mental Model — One Sentence

> Nx is the manager that knows every project in the monorepo, understands how they connect, runs only what needs to run, and scaffolds new code so you never start from a blank file.

```bash
# ── Generators (scaffold new code) ─────────────────────────────────
pnpm nx g @nx/nest:application ...     # Generate a new NestJS app
pnpm nx g @nx/next:application ...     # Generate a new Next.js app
pnpm nx g @nx/nest:resource <name>     # Generate a full CRUD module in NestJS
pnpm nx g @nx/nest:module <name>       # Generate a single NestJS module
pnpm nx g @nx/nest:service <name>      # Generate a NestJS service
pnpm nx g @nx/nest:controller <name>   # Generate a NestJS controller

# ── Running tasks ───────────────────────────────────────────────────
pnpm nx serve api                      # Start NestJS dev server
pnpm nx serve web                      # Start Next.js dev server
pnpm nx build api                      # Build api for production
pnpm nx test api                       # Run tests for api
pnpm nx lint api                       # Lint api

# ── Running tasks across all projects ───────────────────────────────
pnpm nx run-many --target=build --all
pnpm nx run-many --target=test --all
pnpm nx run-many --target=serve --projects=api,web --parallel

# ── Affected (only changed projects) ────────────────────────────────
pnpm nx affected --target=build
pnpm nx affected --target=test
pnpm nx affected --target=lint

# ── Visualize ───────────────────────────────────────────────────────
pnpm nx graph                          # Open dependency graph in browser

# ── Cache ───────────────────────────────────────────────────────────
pnpm nx reset                          # Clear the local Nx cache

# ── Nx Cloud (skip for now) ─────────────────────────────────────────
pnpm nx connect                        # Connect to remote Nx Cloud cache
```
